# 収益化設計 — エッジの源泉と生存戦略

ARCHITECTURE.md がシステムの「作り方」の正典なら、本書は「何で儲けるか」の正典。
前提となる結論: **RL機構は検証済みで、もう主因ではない（ARCHITECTURE.md §2.7）。
勝敗を決めるのはデータのアルファ量・コスト管理・検証の誠実さ・生存設計の4つ。**

---

## 0. 冷徹な前提（これを飲めないなら実弾を入れない）

1. **合成データの +1496% は実データでは起きない。** 合成ベンチはIC 0.1超の世界。
   実データの現実的なICは 0.02〜0.05。この領域での期待値は
   「コスト後Sharpe 1〜2、年率リターン10〜40%、DD 10〜20%」程度が上限目線。
2. **ゲート1（OOSランクIC ≥ 0.02）が全てに優先する。** Hyperliquid生特徴のみでは
   IC ≈ 0.006 で不合格だった実績がある。オーダーフロー・デリバ特徴込みの
   実データでこれを超えられなければ、下流の全工程は無意味。
3. **敵は予測誤差ではなくコスト。** taker 5bps + スプレッド + インパクトを
   1時間リバランスで払い続けると、弱いエッジは簡単に食い潰される。
   本設計の半分はコストを払わない工夫に費やされる。
4. **バックテストで良かったものの大半は本番で劣化する。** 差を生むのは
   （a）検証時に無意識に犯すデータスヌーピング（b）執行の現実
   （c）レジーム変化。本設計はこの3つそれぞれに対応装置を置く。

---

## 1. エッジの源泉 — 4本柱で分散する

単一のエッジに賭けない。性質の異なる4本を同一ポートフォリオ環境で併走させる。

### 柱1: クロスセクショナル相対価値（主力・市場中立）
15銘柄の「相対的な」強弱に賭ける。方向性ベータを取らないため
相場全体の暴落と独立に機能し、ICの再現性が最も高い領域。
- 特徴: 24hランク・ret_z系のクロスセクショナルz・BTC相対
- 既存実装: `csz_*` 特徴、Ridge教師（市場中立教師）

### 柱2: キャリー（funding捕捉・構造的エッジ）
Perpのfundingは「予測」ではなく「構造」。funding正の銘柄をショート、
負をロングに傾けるだけで期待値が発生する。ICゼロでも成立する唯一の柱。
- 既存実装: 環境がfunding授受を報酬に内包、funding_bps/csz_funding特徴
- 強化候補: 予想funding（premium indexから先読み可能）を特徴に追加

### 柱3: マイクロストラクチャ（オーダーフロー・清算）
600日×15銘柄の`rl_orderflow_1m`と`rl_derivatives`（OI/LS/清算）は
個人が通常処理しないデータ。特に清算カスケード後の平均回帰は
発生メカニズム（強制決済による一時的ミスプライス）が明確で頑健。
- 既存実装: of_imbalance・csz_oi・csz_ls（合成ベンチでIC上位常連）
- 強化候補: 清算notionalスパイク検出→逆張り特徴、OI変化率×価格変化の複合

### 柱4: 時系列モメンタム（ベータ捕捉・保険）
純粋なトレンド相場でB&Hに負けない装置。トレンドゲート合格時のみ
`ts_momentum_teacher`が方向性を注入する（既存実装・偽陽性0確認済み）。

**RLの役割はこの4本を「コスト・リスク制約下で1つのウェイトベクトルに合成する」こと。**
各柱の配合比を人間が決めるのではなく、レジームと不確実性に応じて方策が決める。

## 2. コストを払わない設計（防御こそ最大の攻撃）

| 装置 | 効果の実測 | 状態 |
|---|---|---|
| EMA平滑+no-tradeバンド（後処理） | 回転率-71%・リターン+307%→+653% | 実装済み・既定 |
| 回転率ペナルティ（報酬内） | 過剰取引の学習時抑制 | 実装済み |
| decision_every（低頻度化） | 弱シグナル時のコスト削減 | 実装済み・オプション |
| **学習時コスト1.5倍** | 本番より厳しい環境で学習し安全マージン確保 | 未導入・推奨 |
| **15m分割執行**（v1執行層の統合） | 大口リバランスのインパクト削減 | ロードマップ#4 |
| **リプレイ較正**（M1） | バー近似と実約定の乖離を±0.3 Sharpeで監視 | 実装済み・運用へ組込む |

学習時コスト1.5倍は `--phase train` に `cost_multiplier=1.5` を渡すだけで実現可能。
「コスト2倍でも中央値プラス」のwfゲートと合わせ、コスト見積り誤りへの二重防御になる。

## 3. 不確実性で賭け金を変える（実証済みの増幅装置)

弱シグナル領域（=実データの現実）で最も効いた仕組みをフル活用する:

- **3シードアンサンブル + 不一致度グロス縮小**: 弱シグナルで +26.6%→+81.5%
  （Sharpe 4.7→13.9）。実データでは必須、`--ensemble 3` を常用
- **8状態Regime FSM + 専門家ルーティング**: レジームごとに別方策。
  データが十分溜まった銘柄でのみ有効化（プール不足時はgeneralist代替が自動で効く）
- **ボラターゲット（年率0.5）+ DDデリスク（10%開始）**: 静的サイジングではなく
  実現ボラとDDに応じた動的レバレッジ。fractional Kelly の実装的等価物

## 4. 検証のはしご — 嘘を排除する順序

各段の合格なしに次へ進まない。**このはしごを飛ばすことが最大のリスク要因。**

```
1. ゲート1     OOSランクIC ≥ 0.02（オーダーフロー込み実データで）
2. wf          3fold×3seed×コスト2倍の中央値リターン > 0
3. bootstrap   対trend_followingのSharpe差 p < 0.05（run_pipeline --require-significant）
4. リプレイ     バーsim vs 実約定リプレイのSharpe差 ±0.3以内（M1）
5. 紙上運用     最低2週間。/api/signal/latest のシグナルを記録し
               バックテスト予測と実際の紙上PnLの乖離を毎日突合
6. 実弾（極小）  総資産の1%以下・失っても痛くない額から
```

**多重比較の掟**: wf・bootstrapの結果を見て設定を変える行為は1プロジェクトで
数回まで。試行のたびに `docs/model_decision_log.md` に記録し、
最終判断は一度も触っていない未来データ（紙上運用）でのみ行う。

## 5. 生存設計 — 撤退基準を先に決める（ここが最重要）

エッジは腐る。腐ったことを最速で検知し、損失を限定する装置:

### 自動（実装済み・結線して常時稼働させる）
| 装置 | 発火条件 | 動作 |
|---|---|---|
| guardrails | データ鮮度劣化・NaN・日次損失5%・DD20% | フラット化 |
| drift_monitor（M2） | 特徴PSI/KS・予測分布の閾値超過 | アラート+ポジション縮小 |
| pre_trade_risk | レバレッジ・単一銘柄比率・notional超過 | 発注ブロック |
| model_registry（M4） | — | 1コマンドで前バージョンへロールバック |

### 手動（先に紙に書く。破ったら退場）
- **キルライン1**: 実弾運用の累積PnLが、バックテストのブートストラップ
  下側95%CIを30営業日連続で下回ったら停止・全見直し
- **キルライン2**: 最大DDがバックテスト最大DDの1.5倍に達したら即フラット
- **キルライン3**: ゲート1のICが月次再計測で2ヶ月連続 0.02 未満なら運用停止
- **増資規律**: 3ヶ月ごとの見直しでのみ増額。1回の増額は2倍まで

### 週次再学習ループ（チャンピオン/チャレンジャー方式）
```
毎週: fetch(差分) → 品質ゲート → ゲート1再計測 → 再学習
      → 新旧モデルをOOSシャドー比較 → 勝った場合のみ registry activate
      → 負けたら現行維持（モデルの鮮度より安定を優先）
```

## 6. 実行計画（優先順・具体コマンド）

### Phase A: 決定的実験（今すぐ・バックフィル完了後）
600日×15銘柄のオーダーフロー込みデータでゲート1を通るか。**ここで全てが決まる。**
```bash
uv run python scripts/run_pipeline.py --source postgres \
    --timesteps 500000 --ensemble 3 --folds 3 --n-seeds 3 \
    --require-significant --pg-source binance
```
- IC ≥ 0.02 → Phase Bへ
- IC < 0.02 → 特徴を強化して再試行（多ホライズンz 60h/120h、清算スパイク、
  予想funding）。3回試して届かなければ「1hポートフォリオ配分に個人が取れる
  エッジは現状ない」という結論を受け入れ、時間軸か市場を変える

### Phase B: 強化（ゲート1合格後のみ）
1. 学習時コスト1.5倍の導入とwf再検証
2. `--phase regime` でレジーム特化がgeneralistを上回るか検証（上回った場合のみ採用）
3. 15m分割執行の統合（大口リバランス時のみ）

### Phase C: 実戦（Phase B完了後のみ）
1. 紙上運用2〜4週間（乖離分析を毎日）
2. リプレイ較正で執行前提を最終確認
3. 極小実弾 + キルライン運用 + 週次再学習ループ稼働
4. Shadow → Canary → Production ゲート（deploy.yml）で段階昇格

## 7. やらないことリスト（誘惑への防壁）

- **より複雑なRLアルゴリズム探し**（SAC/TD3/Transformer方策等）: ボトルネックは
  そこにない。PPO+アンサンブルで機構は足りていることが実証済み
- **1分足での高頻度化**: 個人のインフラ・手数料水準では執行で負ける
- **wfを何十回も回して良い設定を探す**: それは未来データへの過学習
- **ゲート不合格のまま「もう少しで動きそう」と実弾投入**: 全てのゲートは
  過去の失敗の血で書かれている
- **含み損時の裁量介入**: キルライン以外の裁量停止・再開はしない。
  するならその瞬間からそれは別の（検証されていない）戦略
